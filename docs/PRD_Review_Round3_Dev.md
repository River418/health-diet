### 2.5 第7章 7.6节 数据库设计

| 评估项 | 评估结果 | 说明 |
|--------|----------|------|
| 表结构设计 | ✅ 合理 | 符合第三范式，字段定义清晰 |
| 多语言支持 | ✅ 完善 | 使用`name_zh_cn`等后缀区分语言 |
| JSON字段使用 | ⚠️ 需注意 | crowd_tags/efficacy_tags使用JSON，查询性能需关注 |
| 索引设计 | ⚠️ 需优化 | 部分索引为伪代码（如CAST语法），需按实际数据库调整 |

**数据库索引优化建议**:

```sql
-- 当前PRD中的索引（MySQL 8.0+支持）
KEY `idx_crowd_tags` ((CAST(`crowd_tags` AS CHAR(64) ARRAY)))

-- 建议改为（兼容性更好）
-- 方式1: 使用虚拟列（MySQL 5.7+）
ALTER TABLE recipes ADD COLUMN crowd_tag_1 VARCHAR(64) 
  GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(crowd_tags, '$[0]'))) STORED;
CREATE INDEX idx_crowd_tag_1 ON recipes(crowd_tag_1);

-- 方式2: 使用全文索引（适合多值搜索）
CREATE FULLTEXT INDEX idx_crowd_tags ON recipes(crowd_tags);

-- 方式3: 拆分为关联表（性能最佳，推荐）
CREATE TABLE recipe_tags (
  recipe_id BIGINT,
  tag_type VARCHAR(20), -- 'crowd' | 'efficacy' | 'solar_term'
  tag_value VARCHAR(64),
  PRIMARY KEY (recipe_id, tag_type, tag_value),
  KEY idx_tag_type_value (tag_type, tag_value)
);
```

## 3. 技术风险点

### 3.1 高风险

| 风险项 | 风险描述 | 影响 | 缓解措施 |
|--------|----------|------|----------|
| **JSON字段查询性能** | MySQL/PostgreSQL的JSON字段索引支持有限，多标签筛选可能全表扫描 | 列表查询响应慢 | 方案1: 使用关联表存储标签关系（推荐）<br>方案2: 使用Elasticsearch做搜索层 |

### 3.2 中风险

| 风险项 | 风险描述 | 影响 | 缓解措施 |
|--------|----------|------|----------|
| **视频存储成本** | 视频CDN流量费用较高，初期预算可能不足 | 运营成本超支 | 制定视频压缩策略（720p优先）<br>设置CDN缓存TTL为7天 |
| **内容审核合规** | UGC内容需100%审核通过，人工审核可能成为瓶颈 | 内容发布延迟 | 接入第三方内容审核API（阿里云/腾讯云）<br>设置机器审核+人工抽检模式 |
| **多端数据同步** | 微信/抖音双平台用户数据关联逻辑复杂 | 数据不一致 | 使用手机号作为关联键<br>设计完善的冲突解决策略 |

### 3.3 低风险

| 风险项 | 风险描述 | 影响 | 缓解措施 |
|--------|----------|------|----------|
| **消息队列选型** | RabbitMQ vs Kafka选型未定 | 后期迁移成本 | 抽象消息队列接口层，便于后期切换 |
| **多语言维护成本** | 3种语言×大量文案，维护成本高 | 翻译延迟 | 使用i18n管理平台（如Crowdin）<br>优先保证简中，其他语言可P1阶段补充 |

### 4.2 数据库优化建议

**推荐方案**: 使用关联表存储标签关系

```sql
-- 标签关联表（替代JSON字段）
CREATE TABLE recipe_tag_relations (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  recipe_id BIGINT NOT NULL,
  tag_type VARCHAR(20) NOT NULL COMMENT 'crowd|efficacy|solar_term|ingredient',
  tag_id VARCHAR(64) NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  
  UNIQUE KEY uk_recipe_tag (recipe_id, tag_type, tag_id),
  KEY idx_tag_type_id (tag_type, tag_id),
  KEY idx_recipe_id (recipe_id),
  
  FOREIGN KEY (recipe_id) REFERENCES recipes(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 标签字典表
CREATE TABLE tags (
  id VARCHAR(64) PRIMARY KEY,
  type VARCHAR(20) NOT NULL,
  name_zh_cn VARCHAR(50) NOT NULL,
  name_zh_tw VARCHAR(50),
  name_en VARCHAR(50),
  sort_order INT DEFAULT 0,
  is_active TINYINT DEFAULT 1,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**优势**:
- 查询性能高（B+树索引）
- 支持复杂的多标签组合查询
- 便于统计各标签使用频次
