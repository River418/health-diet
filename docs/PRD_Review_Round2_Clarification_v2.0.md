### 2.2 ADMIN-I01: Tab切换状态保存逻辑

**问题描述**: 食谱编辑页的 Tab 切换（基本信息/食材清单/制作步骤/其他设置）时，未定义状态保存逻辑，可能导致用户数据丢失。

**澄清方案**:

#### 2.2.1 状态保存策略
| 场景 | 行为 | 说明 |
|-----|------|------|
| Tab 切换 | 自动保存草稿 | 切换时自动保存当前 Tab 数据到 localStorage |
| 页面刷新 | 恢复草稿 | 从 localStorage 恢复未提交的编辑数据 |
| 离开页面 | 提示保存 | 检测到未保存数据时弹出确认对话框 |
| 提交保存 | 清除草稿 | 成功保存后清除 localStorage 中的草稿 |

#### 2.2.2 草稿保存机制
```typescript
// 草稿数据结构
interface RecipeDraft {
  recipeId?: number          // 编辑时为 ID，新增时为 undefined
  currentTab: string         // 当前激活的 Tab
  basicInfo: {
    name: string
    description: string
    coverImage: string
    crowdTags: string[]
    efficacyTags: string[]
    solarTermTags: string[]
  }
  ingredients: Ingredient[]
  steps: Step[]
  otherSettings: {
    status: number
    tips: string[]
  }
  lastSavedAt: string        // 最后保存时间
}

// 自动保存触发时机
1. Tab 切换时
2. 表单字段失去焦点后 3 秒（防抖）
3. 用户点击「保存草稿」按钮时
```

#### 2.2.3 草稿恢复流程
```
【进入编辑页】
    │
    ▼
【检查是否存在草稿】
    │
    ├── 是 → 【检查草稿时间】
    │            │
    │            ├── 24小时内 → 【提示恢复草稿】
    │            │                      │
    │            │                      ├── 用户确认 → 【加载草稿数据】
    │            │                      └── 用户取消 → 【加载原始数据】
    │            │
    │            └── 超过24小时 → 【自动清除过期草稿】
    │
    └── 否 → 【加载原始数据】
```

#### 2.2.4 用户提示文案
| 场景 | 提示文案 |
|-----|---------|
| 草稿恢复提示 | "检测到您有未提交的编辑内容（保存于 {time}），是否恢复？" |
| 离开确认提示 | "您有未保存的更改，确定要离开吗？" |
| 自动保存成功 | "草稿已自动保存"（Toast，2秒消失） |
| 草稿清除提示 | "编辑内容已成功提交，草稿已清除" |

**PRD 更新位置**: 第3章 3.3.3 节「食谱表单页」后新增「Tab 状态保存逻辑」

### 2.3 ADMIN-I03: 删除二次确认流程

**问题描述**: 分类、信息流、食谱的删除操作未定义二次确认流程，存在误操作风险。

**澄清方案**:

#### 2.3.1 删除确认流程
```
【用户点击删除按钮】
    │
    ▼
【弹出确认对话框】
    │
    ├── 用户点击「取消」→ 【关闭对话框，无操作】
    │
    └── 用户点击「确认删除」→ 【执行删除】
                │
                ▼
        【删除成功】→ 【显示成功提示，刷新列表】
                │
                ▼
        【删除失败】→ 【显示错误提示，保留对话框】
```

#### 2.3.2 确认对话框设计
```
┌─────────────────────────────────────────┐
│  ⚠️ 确认删除                            │
├─────────────────────────────────────────┤
│                                         │
│  确定要删除「{itemName}」吗？           │
│                                         │
│  此操作不可撤销，相关数据将被永久删除。 │
│                                         │
│  [取消]              [确认删除]         │
│                                         │
└─────────────────────────────────────────┘
```

#### 2.3.3 不同模块的删除提示
| 模块 | 提示内容 | 特殊处理 |
|-----|---------|---------|
| 分类 | "确定要删除分类「{name}」吗？该分类下的 {count} 个食谱将变为未分类状态。" | 显示关联食谱数量 |
| 信息流 | "确定要删除内容「{title}」吗？" | 标准提示 |
| 食谱 | "确定要删除食谱「{name}」吗？该食谱有 {favoriteCount} 人收藏，{commentCount} 条评论。" | 显示收藏和评论数量 |

#### 2.3.4 批量删除
```
【用户选择多条记录】
    │
    ▼
【点击批量删除】
    │
    ▼
【弹出确认对话框】
    │
    ▼
【确认后批量执行】
    │
    ├── 全部成功 → 【显示成功提示，刷新列表】
    │
    └── 部分失败 → 【显示失败列表，提供重试】
```

批量删除确认文案：
"确定要删除选中的 {count} 项吗？此操作不可撤销。"

**PRD 更新位置**: 第3章各模块「页面设计」后新增「删除操作规范」

### 2.4 ADMIN-V04: 响应式适配规则

**问题描述**: 管理后台未定义响应式适配规则，仅提到「响应式布局适配」，缺少断点定义和适配策略。

**澄清方案**:

#### 2.4.1 断点定义
| 断点名称 | 宽度范围 | 设备类型 |
|---------|---------|---------|
| xs | < 576px | 手机（不支持） |
| sm | 576px - 767px | 小平板（不推荐） |
| md | 768px - 991px | 平板/小笔记本 |
| lg | 992px - 1199px | 笔记本 |
| xl | 1200px - 1599px | 桌面显示器 |
| xxl | ≥ 1600px | 大屏显示器 |

#### 2.4.2 最小支持尺寸
**管理后台最小支持宽度**: 1280px  
**推荐分辨率**: 1920×1080 及以上  
**说明**: 管理后台面向运营人员，主要在桌面环境使用，不对手机端做专门适配。

#### 2.4.3 各断点适配策略

**md (768px - 991px) - 平板**
- 侧边栏：收起为图标模式（宽度 80px）
- 表格：横向滚动，关键列固定
- 表单：单列布局
- 操作按钮：图标 + Tooltip

**lg (992px - 1199px) - 笔记本**
- 侧边栏：展开（宽度 220px）
- 表格：自适应列宽，非关键列可隐藏
- 表单：双列布局
- 搜索区域：折叠为「高级筛选」

**xl (1200px - 1599px) - 桌面**
- 侧边栏：展开（宽度 250px）
- 表格：完整显示所有列
- 表单：双列布局
- 搜索区域：完全展开

**xxl (≥ 1600px) - 大屏**
- 侧边栏：展开（宽度 250px）
- 表格：完整显示，适当增加列宽
- 表单：三列布局（部分页面）
- 内容区最大宽度：1400px，居中显示

### 2.5 ACC-01: 大字模式技术兼容性

**问题描述**: PRD 中提到「支持系统字体大小跟随」，但未明确技术实现方案和兼容性处理。

**澄清方案**:

#### 2.5.1 技术实现方案

**方案 A: CSS 变量 + Class 切换（推荐）**
```scss
// 基础字体大小
:root {
  --font-size-xs: 12px;
  --font-size-sm: 14px;
  --font-size-base: 16px;
  --font-size-lg: 18px;
  --font-size-xl: 20px;
  --font-size-2xl: 24px;
}

// 大字模式
.font-large {
  --font-size-xs: 14px;
  --font-size-sm: 16px;
  --font-size-base: 18px;
  --font-size-lg: 20px;
  --font-size-xl: 22px;
  --font-size-2xl: 26px;
}

// 使用变量
.text-sm { font-size: var(--font-size-sm); }
.text-base { font-size: var(--font-size-base); }
.text-lg { font-size: var(--font-size-lg); }
```

**方案 B: 动态计算（备用）**
```typescript
// 根据系统字体大小动态计算
const getFontSize = (baseSize: number): number => {
  const systemFontScale = Taro.getSystemInfoSync().fontSizeSetting || 1
  return baseSize * systemFontScale
}
```
