# 平台感知登录设计

**日期：** 2026-04-05  
**目标：** 在不引入单独登录页供用户选择平台的前提下，用户在未登录状态触发需要鉴权的操作时，系统自动根据当前运行端发起微信或抖音授权登录，并在成功后继续原操作。

## 背景

当前前端是一个 Taro 多端项目，已支持：

- 微信小程序构建：`build:weapp`
- 抖音小程序构建：`build:tt`

当前登录页 `frontend/src/pages/login/index.vue` 仍采用“微信按钮 / 抖音按钮”显式选择模式；同时多个业务页面在未登录时会直接跳转到 `/pages/login/index`。这与期望的体验不一致。

## 设计目标

1. 用户不再进入独立登录页选择登录方式。
2. 未登录用户触发鉴权操作时，系统自动识别当前平台并直接发起对应授权。
3. 登录成功后继续执行用户原本的动作。
4. 所有业务页面统一复用一套登录守卫逻辑，避免分散实现。
5. 不引入 H5 容器识别或 UA 判断，优先使用 Taro 当前运行端能力。

## 非目标

以下内容不在本次范围内：

- H5 页面在微信/抖音 App 内的 UA 探测登录
- 小程序 `web-view` 内嵌 H5 授权桥接
- 新增第三方登录平台
- 重构后端授权协议

## 方案概述

采用“统一登录服务 + 统一登录守卫”的结构：

1. 前端新增统一登录服务，负责：
   - 判断当前运行端
   - 调用对应平台的 `Taro.login`
   - 请求现有后端登录接口
   - 写入 `userStore` 与本地存储
2. 前端新增 `requireLogin()` 守卫，负责：
   - 已登录时直接放行
   - 未登录时触发统一登录服务
   - 登录成功后继续执行原始业务动作
3. 业务页面不再跳转 `/pages/login/index`，而改为统一调用 `requireLogin()`。

## 平台判定策略

本项目是 Taro 多端构建，不需要用 User-Agent 猜测平台。优先按 Taro 运行环境判定：

- 微信小程序端：走微信登录流程
- 抖音小程序端：走抖音登录流程
- 其他端：提示“当前环境暂不支持快捷登录”

这样可以避免误判，并与当前仓库的构建形态保持一致。

## 登录流程设计

### 1. 微信小程序

1. 调用 `Taro.login()` 获取 `code`
2. 调用现有接口 `POST /auth/wechat/login`
3. 后端继续使用现有 `jscode2session` 流程换取 `openid`
4. 前端写入：
   - `token`
   - `refreshToken`
   - `userInfo`
5. 返回登录成功结果

### 2. 抖音小程序

1. 调用 `Taro.login({ force: true })` 获取 `code` 与可能的 `anonymousCode`
2. 调用现有接口 `POST /auth/douyin/login`
3. 后端继续使用现有抖音换取会话能力
4. 前端写入：
   - `token`
   - `refreshToken`
   - `userInfo`
5. 返回登录成功结果

### 3. 非受支持端

如果当前端既不是微信小程序也不是抖音小程序：

- 不发起登录请求
- 直接提示当前环境不支持快捷登录
- 返回失败态给调用方

## 前端模块设计

建议新增如下模块：

### `frontend/src/services/auth/login.ts`

职责：

- 封装平台识别
- 执行微信/抖音登录流程
- 调用现有 `frontend/src/api/auth.ts`
- 统一处理成功态与失败态
- 写入 `userStore` 和本地缓存

建议导出：

- `loginByCurrentPlatform()`

### `frontend/src/services/auth/requireLogin.ts`

职责：

- 统一登录守卫
- 支持包装业务动作
- 确保登录成功后继续原操作

建议导出：

- `requireLogin()`：仅保证当前已登录
- `withLoginGuard(action)`：登录成功后执行传入动作

### `frontend/src/pages/login/index.vue`

本页不再作为主登录入口。处理建议：

- 第一阶段：保留为兜底页，内部直接调用统一登录服务
- 第二阶段：当所有入口都完成迁移后，可评估删除

## 业务接入范围

首批接入以下已发现的未登录跳转点：

- `frontend/src/pages/recipe/detail/index.vue`
  - 收藏
  - 评论输入
- `frontend/src/pages/recipe/comments/index.vue`
  - 点赞评论
  - 回复评论
  - 发表评论
- `frontend/src/pages/profile/settings/index.vue`
  - 当前退出逻辑中的回到登录页行为需要重新审视，避免把登录页继续当成主入口
- `frontend/src/pages/user/index.vue`
  - 增加缺失的 `handleLoginClick`
  - 点击未登录文案时直接触发统一登录

后续可以继续全局搜索 `/pages/login/index`，将其余入口统一迁移。

## 状态与存储

当前 `userStore` 已有：

- `setToken`
- `setUserInfo`
- `logout`

本次需要补齐/确认：

- 登录成功后同步存储 `refreshToken`
- 退出登录时一并清除 `refreshToken`
- 统一登录服务不要重复散落写本地存储逻辑，尽量通过 store 收口

## 错误处理

### 用户可见错误

- 平台不支持：提示当前环境暂不支持快捷登录
- 授权失败：提示授权失败，请重试
- 接口异常：提示网络或服务异常

### 行为约束

- 登录失败时不污染现有登录态
- 登录中防重入，避免重复点按触发多次授权
- 业务动作只在登录成功后执行

## 并发与重入

统一登录守卫需要保证：

- 同一时刻仅允许一个登录流程进行中
- 多处业务入口同时触发登录时，复用同一个登录过程或直接拦截重复发起

这样可以避免多个 `Taro.login()` 并发造成状态混乱。

## 测试设计

### 单元测试

覆盖以下核心行为：

1. 微信运行端时调用微信登录 API
2. 抖音运行端时调用抖音登录 API
3. 非支持端返回失败并提示
4. `requireLogin()` 在已登录时直接通过
5. `requireLogin()` 在未登录时触发登录并继续动作
6. 登录失败时不执行原动作

### 集成验证

至少验证：

- 微信小程序真机 / 开发者工具登录成功
- 抖音小程序真机 / 开发者工具登录成功
- 收藏、评论、个人页登录触发点行为正常

## 实施顺序

1. 抽离统一登录服务
2. 增加统一登录守卫
3. 改造 `userStore` 的 token / refreshToken 收口
4. 将已知业务页面入口切换为 `requireLogin()`
5. 将登录页调整为兜底实现
6. 补测试并完成多端验证

## 风险与注意事项

1. `frontend/src/pages/user/index.vue` 当前模板引用了 `handleLoginClick`，但脚本内未定义；改造时应一起修复。
2. `frontend/src/api/auth.ts` 当前返回类型声明是 `LoginResult`，但 `request.ts` 实际返回包裹结构为 `ApiResponse<T>`；实现时需要统一类型使用方式，避免继续把响应体当裸数据使用。
3. `refreshToken` 当前刷新流程依赖本地存储，但登录成功流程未看到对应存储收口，实施时需要一并补齐。
4. 当前后端接口已经具备基础平台登录能力，优先复用，不要在本次设计里引入新的授权协议。

## 验收标准

满足以下条件即视为完成：

1. 用户在微信小程序中点击任一需要登录的操作时，直接触发微信授权登录。
2. 用户在抖音小程序中点击任一需要登录的操作时，直接触发抖音授权登录。
3. 不再要求用户进入单独登录页选择平台。
4. 登录成功后继续执行用户原操作。
5. 登录失败时只提示错误，不破坏当前页面状态。
6. 已知业务入口不再直接跳转 `/pages/login/index`。
